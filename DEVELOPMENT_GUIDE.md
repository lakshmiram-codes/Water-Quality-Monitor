# WaterWatch — Development Guide

This guide covers local setup, the API reference, how the alerting and predictive
logic works, testing, and deployment.

---

## 1. Prerequisites

| Tool           | Version (tested against) |
|----------------|---------------------------|
| Python         | 3.12                      |
| Node.js        | 20.x                      |
| PostgreSQL     | 16.x                      |
| Docker + Compose (optional but recommended) | latest |

## 2. Local setup — with Docker (recommended)

From the project root:

```bash
cp backend/.env.example backend/.env   # adjust SECRET_KEY etc. if you like
docker compose up --build
```

What this does:
1. Starts a PostgreSQL 16 container with a persistent volume.
2. Builds and starts the FastAPI backend, which on boot runs:
   - `alembic upgrade head` (creates all tables)
   - `python seed.py` (loads demo users, stations, readings, a report, and an alert —
     skips automatically if data already exists)
   - `uvicorn app.main:app` on port 8000
3. Builds and starts the Vite dev server for the React frontend on port 5173.

Visit **http://localhost:5173** and log in with any demo account from the README.

To stop: `docker compose down`. To wipe the database and reseed: `docker compose down -v`.

## 3. Local setup — without Docker

### Backend

```bash
cd backend
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt

# Create a local Postgres database + user matching .env, e.g.:
#   createuser waterwatch -P   (set password "waterwatch")
#   createdb waterwatch -O waterwatch

cp .env.example .env             # edit DATABASE_URL / SECRET_KEY as needed
alembic upgrade head             # create tables
python seed.py                   # load demo data
uvicorn app.main:app --reload --port 8000
```

The API is now live at `http://localhost:8000`, with interactive docs at
`http://localhost:8000/docs` (Swagger UI, auto-generated by FastAPI).

### Frontend

```bash
cd frontend
npm install
cp .env.example .env             # VITE_API_BASE_URL=/api is correct for local dev
npm run dev
```

Vite's dev server proxies `/api` and `/uploads` to `http://localhost:8000` (see
`vite.config.js`), so the frontend and backend can run on separate ports without CORS
issues in local dev.

## 4. Environment variables (backend/.env)

| Variable | Purpose | Default |
|---|---|---|
| `DATABASE_URL` | SQLAlchemy connection string | `postgresql+psycopg2://waterwatch:waterwatch@localhost:5432/waterwatch` |
| `SECRET_KEY` | JWT signing secret — **change in production** | dev placeholder |
| `ALGORITHM` | JWT algorithm | `HS256` |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | Access token lifetime | `30` |
| `REFRESH_TOKEN_EXPIRE_DAYS` | Refresh token lifetime | `7` |
| `CORS_ORIGINS` | Comma-separated allowed origins | `http://localhost:5173,http://localhost:3000` |
| `UPLOAD_DIR` | Where report photos are saved | `./uploads` |
| `EPA_API_KEY`, `EPA_API_BASE_URL` | US EPA / Water Quality Portal credentials | empty |
| `WHO_API_BASE_URL` | WHO data source base URL | empty |
| `CPCB_API_KEY`, `CPCB_API_BASE_URL` | CPCB India credentials | empty |
| `PREDICTIVE_LOOKBACK_READINGS` | How many past readings the risk model uses | `10` |

**None of the external API variables are required to run the app.** If they're
blank, station "sync" calls automatically fall back to a realistic mock-data
generator (see section 7), so the map, alerts, and analytics all work out of the box.

## 6. Database schema

Implemented exactly per the original spec, via SQLAlchemy models in
`backend/app/models/` and the Alembic migration in
`backend/alembic/versions/0001_initial_schema.py`:

- **Users** — id, name, email (unique), password (bcrypt hash), role (enum), location, created_at
- **Reports** — id, user_id (FK), photo_url, location, description, water_source, status (enum), station_id (FK), created_at
- **WaterStations** — id, name, location, latitude, longitude, managed_by, created_at
- **StationReadings** — id, station_id (FK), parameter (enum), value, recorded_at
- **Alerts** — id, alert_type (enum), message, location, station_id (FK), report_id (FK), issued_at
- **Collaborations** — id, ngo_id (FK to Users), project_name, station_id (FK), contact_email, created_at

To create a new migration after changing a model:

```bash
cd backend
alembic revision --autogenerate -m "describe your change"
alembic upgrade head
```

## 7. External water-quality API adapters

Location: `backend/app/services/external_apis.py`.

Each of `EPAAdapter`, `WHOAdapter`, and `CPCBAdapter` implements a common interface:
`is_configured()` and `fetch_readings(station)`. `fetch_readings_for_station()` tries
each adapter in turn.

- **If real credentials are set** (`EPA_API_KEY`, `WHO_API_BASE_URL`,
  `CPCB_API_KEY`+`CPCB_API_BASE_URL`), the adapter calls the real endpoint.
- **If not configured, or the live call fails for any reason**, the adapter
  transparently falls back to `_fetch_mock()`, which generates one plausible
  reading per tracked parameter within realistic ranges. This means the app is
  always demoable and never breaks due to an external outage or missing key.

To go live with a real source:
1. Fill in the relevant `.env` variables.
2. If the real API's response shape differs from `{"parameter", "value",
   "recorded_at"}`, adjust `_fetch_live()` in that adapter to map fields —
   the rest of the app (alerts, analytics, map) doesn't need to change.

Station readings are pulled on demand via `POST /api/stations/{id}/sync` (exposed
in the UI as a "Sync latest readings" button for authority/admin users). For a
production deployment, wire this same call into a scheduled job (cron, Celery
beat, or a cloud scheduler) to poll periodically instead of on demand.

## 8. Alert-triggering logic

Location: `backend/app/services/alert_engine.py`.

Two triggers, both run automatically (no manual step required):

1. **`evaluate_reading(db, reading)`** — runs whenever a `StationReading` is created
   (via manual entry or a station sync). Compares the value against a simplified
   safe range per parameter (`SAFE_RANGES`); out-of-range values create a
   `contamination` alert.
2. **`evaluate_report(db, report)`** — runs whenever a citizen `Report` is created.
   Scans the description/water_source text for keyword groups: contamination
   indicators (smell, discoloration, dead fish, etc.) or boil-notice indicators
   (main break, low pressure, bacteria, etc.) and creates the matching alert type.

Authorities/admins can also issue alerts manually via `POST /api/alerts` for
situations the automatic rules don't cover (e.g., a planned outage).

The safe-range thresholds and keyword lists are simplified for demonstration —
see "Known limitations" below before using this in a real regulatory context.

## 9. Predictive analytics baseline

Location: `backend/app/services/predictive.py`, exposed at
`GET /api/analytics/predict/{station_id}/{parameter}`.

Method: for the most recent `PREDICTIVE_LOOKBACK_READINGS` readings of a given
station/parameter —
1. Compute the mean/standard deviation of all but the latest reading (the "baseline").
2. Compute how many standard deviations the latest reading is from that baseline
   (z-score).
3. Compute a simple least-squares trend slope across the whole window, normalized
   by the mean, to capture directional drift even before a single reading crosses
   a hard threshold.
4. Combine both into a 0-1 `risk_score` (weighted 70% z-score, 30% trend), bucketed
   into `low` / `medium` / `high` for the UI.

This is a transparent, auditable statistical baseline chosen specifically so it's
easy to reason about and to replace. It is **not** a trained machine-learning model.

## 10. Testing

Backend tests use an in-memory SQLite database (via a fixture in
`backend/tests/conftest.py`) so they never touch your real Postgres instance.

```bash
cd backend
source .venv/bin/activate   # if not already active
pytest -v
```

Covers: registration/login/duplicate-email/auth-required (`test_auth.py`), report
creation + role-based moderation + keyword-triggered alerts (`test_reports.py`),
and station/reading creation + role guards + threshold-triggered alerts
(`test_stations_and_readings.py`).

## 11. Deployment notes

A reasonable low-cost production layout:

- **Database**: managed PostgreSQL (Render, Railway, Supabase, or RDS).
- **Backend**: Render/Railway/Fly.io running the provided `backend/Dockerfile`,
  or any host that runs `uvicorn app.main:app`. Run `alembic upgrade head` as a
  release/pre-deploy step (not on every container start, to avoid race conditions
  with multiple instances).
- **Frontend**: Vercel or Netlify, building with `npm run build` and serving the
  `dist/` output. Set `VITE_API_BASE_URL` to your deployed backend's URL
  (e.g., `https://api.yourdomain.com/api`).
- **File uploads**: the local-disk upload approach in `reports.py` works for a
  single backend instance; for multi-instance deployments, swap it for S3-compatible
  object storage (change `create_report`'s file-write to an upload call, and store
  the resulting URL in `photo_url` — no schema change needed).

Required production environment variables: `DATABASE_URL`, `SECRET_KEY` (generate
a long random value — do not reuse the dev default), `CORS_ORIGINS` (your deployed
frontend origin), and optionally the external API credentials from section 4.

## 12. Known limitations & suggested next steps

- **Mock external data by default.** Real EPA/WHO/CPCB integration requires
  credentials and, for some sources, mapping their actual response schema
  (see section 7) — the adapters are structured so this is a localized change.
- **Alert thresholds are illustrative**, not official regulatory limits. Before
  any real-world use, replace `SAFE_RANGES` in `alert_engine.py` with
  jurisdiction-specific standards (e.g., EPA/WHO published limits) and have them
  reviewed by a domain expert.
- **Predictive model is a statistical baseline**, not a trained ML model. A
  natural next step is collecting enough historical labeled data (confirmed
  contamination events) to train a real classifier, while keeping this
  statistical version as a fast, explainable fallback.
- **No email/SMS notifications yet** — `Alert` records are created and shown in
  the dashboard/UI; wiring them to an email (e.g., SES, Postmark) or SMS
  (e.g., Twilio) provider is a contained addition to `alert_engine.py`.
- **No rate limiting or account email verification** — recommended additions
  before any public-facing production deployment.
- **Single-instance file uploads** — see the object-storage note in section 11
  for scaling beyond one backend instance.
